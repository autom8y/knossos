# Agent Migration Guide

**For**: Rite authors, agent developers, existing agent maintainers
**Requires**: ari CLI with `ari agent` commands available
**Related**: [ADR-0024](../decisions/ADR-0024-agent-factory.md), [CLI reference](../cli/agent-commands.md)

---

## What Changed and Why

Agent files now use a formal YAML frontmatter schema with validation, type classification, and structured metadata. Previously, agents were freeform markdown with minimal frontmatter (`name`, `description`, `tools` as a comma string). The new system adds:

- **JSON Schema validation** (`agent.schema.json`) for frontmatter correctness
- **7 agent types** (`orchestrator`, `specialist`, `reviewer`, `meta`, `designer`, `analyst`, `engineer`)
- **3 archetypes** with template-driven scaffolding and section ownership
- **Workflow metadata** (`upstream`, `downstream`, `produces`)
- **Behavioral contracts** (`must_use`, `must_produce`, `must_not`, `max_turns`)
- **MCP tool references** (`mcp:<server>/<method>`)
- **CLI tooling** for creation, validation, listing, and platform section updates

Existing agents continue to work without modification. Enhanced metadata is optional and can be added incrementally. The two-tier validation system (WARN mode vs. STRICT mode) ensures backward compatibility while enabling stricter CI gates for new agents.

## Before and After: Agent Frontmatter

### Before (Minimal Frontmatter)

```yaml
---
name: technology-scout
description: "Technology horizon specialist who evaluates emerging technologies"
tools: Glob, Grep, Read, Write, WebSearch, WebFetch, TodoWrite, Skill
model: opus
color: orange
---
```

### After (Enhanced Frontmatter)

```yaml
---
name: technology-scout
description: |
  Technology horizon specialist who evaluates emerging technologies
  for competitive advantage. Invoke for tech radar updates and
  adoption recommendations.
role: Evaluates emerging technologies for competitive advantage
type: specialist
tools:
  - Glob
  - Grep
  - Read
  - Write
  - WebSearch
  - WebFetch
  - TodoWrite
  - Skill
model: opus
color: orange
upstream:
  - source: orchestrator
    artifact: research brief
downstream:
  - agent: integration-researcher
    condition: "recommendation in [Adopt, Trial, Assess]"
    artifact: tech assessment
produces:
  - artifact: tech-assessment
    format: markdown
contract:
  must_use:
    - WebSearch
  must_produce:
    - tech-assessment
  must_not:
    - modify source code
    - commit changes
  max_turns: 50
schema_version: "1.0"
---
```

Key differences:

1. `tools` can be a YAML array (recommended) or a comma-separated string (still supported)
2. `type` classifies the agent for archetype behavior
3. `upstream`/`downstream`/`produces` declare workflow position
4. `contract` formalizes behavioral constraints
5. `description` can be multi-line with the `|` YAML literal block scalar

## Creating a New Agent with `ari agent new`

### Step 1: Choose an Archetype

| Archetype | When to Use | Default Tools |
|-----------|-------------|---------------|
| `orchestrator` | Coordinates work across specialists; does not execute work directly | `Read` |
| `specialist` | Executes focused work in a specific domain; produces artifacts | `Bash`, `Glob`, `Grep`, `Read`, `Edit`, `Write`, `TodoWrite`, `Skill` |
| `reviewer` | Reviews work products; approves, rejects, or routes back | `Bash`, `Glob`, `Grep`, `Read`, `Edit`, `Write`, `WebFetch`, `WebSearch`, `TodoWrite`, `Skill` |

### Step 2: Scaffold the Agent

```bash
ari agent new \
  --archetype specialist \
  --rite rnd \
  --name technology-scout \
  --description "Researches emerging technologies and produces evaluation reports"
```

**Verify**: Output shows `Created rites/rnd/agents/technology-scout.md -- fill in author sections marked with TODO`

**Verify**: File exists at `rites/rnd/agents/technology-scout.md`

### Step 3: Fill in Author-Owned Sections

Open the generated file. Look for `<!-- TODO: ... -->` markers. These mark author-owned sections that require your domain-specific content.

For a specialist agent, the author sections are:

| Section | What to Write |
|---------|---------------|
| Core Responsibilities | Primary and secondary functions of this agent |
| Domain Authority | What this agent decides, escalates, and routes |
| What You Produce | Artifacts with format and intended audience |
| Quality Standards | How to verify the agent's work meets quality bar |
| Handoff Criteria | Checklist that must be complete before handoff |

Do **not** modify sections marked as Platform or Derived ownership. These are managed by the platform and will be regenerated by `ari agent update`.

### Step 4: Add Enhanced Frontmatter (Optional)

Add workflow metadata and behavioral contracts to the frontmatter:

```yaml
upstream:
  - source: orchestrator
    artifact: research brief
downstream:
  - agent: integration-researcher
    condition: assessment complete
    artifact: tech assessment
produces:
  - artifact: tech-assessment
    format: markdown
contract:
  must_use:
    - WebSearch
  must_produce:
    - tech-assessment
  must_not:
    - modify source code
```

### Step 5: Validate

```bash
# Default validation (WARN mode)
ari agent validate rites/rnd/agents/technology-scout.md
```

**Verify**: Output shows `PASS` or `WARN` (no `FAIL`)

```bash
# Strict validation (all enhanced fields required)
ari agent validate --strict rites/rnd/agents/technology-scout.md
```

**Verify**: Output shows `PASS` (no `FAIL` or `WARN`)

## Migrating an Existing Agent to Enhanced Frontmatter

If you have an existing agent with minimal frontmatter, you can add enhanced metadata incrementally. No fields need to change; you only add new ones.

### Step 1: Validate Current State

```bash
ari agent validate rites/myrite/agents/my-agent.md
```

**Verify**: Agent passes WARN-mode validation (existing agents should pass). If it fails, fix the reported issues first (usually a missing `name` or `description` field, or a typo in a tool name).

### Step 2: Add `type` Field

Add the type field that best describes your agent's role:

```yaml
type: specialist  # or orchestrator, reviewer, meta, designer, analyst, engineer
```

See the [Type Assignment Guide](#type-assignment-guide) below for which type to choose.

### Step 3: Convert `tools` to Array Format (Optional)

Convert from comma-separated string to YAML array. Both formats work, but the array format is recommended for new and migrated agents.

Before:
```yaml
tools: Bash, Glob, Grep, Read, Edit, Write, TodoWrite, Skill
```

After:
```yaml
tools:
  - Bash
  - Glob
  - Grep
  - Read
  - Edit
  - Write
  - TodoWrite
  - Skill
```

### Step 4: Add Workflow Metadata (Optional)

If the agent participates in a multi-agent workflow:

```yaml
upstream:
  - source: context-architect
    artifact: design document
downstream:
  - agent: documentation-engineer
    condition: implementation complete
    artifact: implementation report
produces:
  - artifact: implementation code
    format: go
```

### Step 5: Add Behavioral Contract (Optional)

```yaml
contract:
  must_use:
    - Read       # Must read before editing
  must_produce:
    - integration tests
  must_not:
    - skip test verification
    - modify files outside scope
  max_turns: 50
```

Note: For `reviewer` type agents, `contract.must_not` is required in strict mode.

### Step 6: Run Platform Section Update

After adding the `type` field, run the update command to regenerate platform and derived sections:

```bash
# Preview what would change
ari agent update --dry-run rites/myrite/agents/my-agent.md
```

**Verify**: Output shows which sections would be regenerated.

```bash
# Apply the update
ari agent update rites/myrite/agents/my-agent.md
```

**Verify**: Output shows `UPDATED` with the number of sections regenerated.

Your author-owned sections are preserved exactly as written. Only platform-owned sections (behavioral constraints, anti-patterns, acid test) and derived sections (tool access, workflow position, skills reference) are regenerated.

### Step 7: Validate Strict

```bash
ari agent validate --strict rites/myrite/agents/my-agent.md
```

**Verify**: Output shows `PASS`

### Rollback

If the update produces undesirable changes, restore from git:

```bash
git checkout -- rites/myrite/agents/my-agent.md
```

## Section Ownership Model

Every section in an archetype template has an ownership designation that controls who writes and updates it.

### Platform Sections

**Owned by**: The Knossos platform (archetype templates)
**Examples**: Behavioral Constraints, The Acid Test, Anti-Patterns, Consultation Protocol (orchestrators), Handling Failures (orchestrators)
**Author action**: Do not modify. These are regenerated by `ari agent update`.

Platform sections contain standard guidance that applies to all agents of an archetype. When the platform updates this guidance (e.g., adding a new anti-pattern), running `ari agent update --all` propagates the change to every agent.

### Author Sections

**Owned by**: The agent author
**Examples**: Core Responsibilities, Domain Authority, What You Produce, Quality Standards, Handoff Criteria, Phase Routing (orchestrators)
**Author action**: Fill these in with domain-specific content. These are never modified by `ari agent update`.

When a new agent is scaffolded, author sections contain `<!-- TODO: hint -->` markers with guidance on what to write.

### Derived Sections

**Owned by**: Generated from frontmatter data
**Examples**: Tool Access (generated from `tools` field), Position in Workflow (generated from `upstream`/`downstream`), Skills Reference
**Author action**: Do not modify. These are regenerated by `ari agent update` based on current frontmatter values.

When you change the `tools` list or `upstream`/`downstream` fields in frontmatter, run `ari agent update` to regenerate derived sections.

### Extra Sections

Sections not defined in the archetype are preserved at the end of the file during `ari agent update`. They are treated as author-owned and never modified or removed.

## Type Assignment Guide

Seven types are available. Choose based on the agent's primary function.

| Type | When to Use | Archetype Template |
|------|-------------|-------------------|
| `orchestrator` | Coordinates multi-phase workflows. Routes work to specialists. Does not execute work directly. Uses only `Read` tool. | orchestrator |
| `specialist` | Executes focused work in a specific domain. Receives prompts, produces artifacts, hands off to downstream agents. The default for most agents. | specialist |
| `reviewer` | Quality gate. Reviews work products against criteria. Provides approve/reject decisions. Must define `contract.must_not` in strict mode. | reviewer |
| `meta` | Platform-level agents that operate across rites (e.g., Moirai for context mutation). Not domain-specific. | specialist (fallback) |
| `designer` | Designs systems, schemas, patterns. Produces design documents, not code. | specialist (fallback) |
| `analyst` | Analyzes problems, traces root causes, produces gap analyses. Investigative rather than constructive. | specialist (fallback) |
| `engineer` | Implements changes with code. Writes tests. Produces working software. | specialist (fallback) |

**Fallback types** (`meta`, `designer`, `analyst`, `engineer`) use the specialist archetype template. The type field carries semantic meaning (for listing, filtering, documentation) but does not change the section layout.

### Decision Flow

1. Does the agent coordinate other agents without doing work itself? --> `orchestrator`
2. Does the agent approve/reject work products? --> `reviewer`
3. Does the agent operate across rites at the platform level? --> `meta`
4. Does the agent primarily produce designs/schemas? --> `designer`
5. Does the agent primarily investigate and analyze? --> `analyst`
6. Does the agent primarily write code and tests? --> `engineer`
7. None of the above? --> `specialist`

## MCP Tool References

Agents can declare MCP (Model Context Protocol) tool dependencies in the `tools` field.

### Format

```
mcp:<server-name>                  # Reference to an MCP server (all tools)
mcp:<server-name>/<tool-name>      # Reference to a specific MCP tool
```

### Examples

```yaml
tools:
  - Read
  - Write
  - Bash
  - mcp:github                     # All tools from the github MCP server
  - mcp:github/create_issue        # Specific tool
  - mcp:github/search_repositories # Specific tool
  - mcp:terraform                  # All tools from terraform MCP server
```

### Pattern Rules

- Server names: lowercase alphanumeric with hyphens (`[a-z0-9-]+`)
- Tool names: lowercase alphanumeric with underscores and hyphens (`[a-z0-9_-]+`)
- Valid: `mcp:github`, `mcp:github/create_issue`, `mcp:go-semantic`
- Invalid: `mcp:GitHub`, `mcp:github/Create_Issue`, `MCP:github`

### Rite Manifest Declaration

For MCP tools to be available in a satellite, the rite manifest must declare the MCP server:

```yaml
# rites/myrite/manifest.yaml
name: myrite
version: "1.0.0"
mcp_servers:
  - name: github
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "${GITHUB_TOKEN}"
  - name: go-semantic
    command: go-semantic-mcp
```

During `ari sync materialize`, these declarations are merged into `settings.local.json`:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}" }
    },
    "go-semantic": {
      "command": "go-semantic-mcp"
    }
  }
}
```

### Validation Behavior

MCP tool references are validated in two stages:

1. **Format validation**: Always checked. Invalid patterns like `MCP:github` or `mcp:GitHub/Create_Issue` produce errors.
2. **Server cross-reference**: Checked when rite context is available (`ari agent validate --rite myrite`). Produces **warnings** (not errors) if the referenced server is not declared in the rite manifest. This is intentional -- satellites may provide their own MCP servers.

## Common Issues and Troubleshooting

### YAML Quoting for Descriptions

Multi-line descriptions must use YAML block scalars:

```yaml
# Correct: literal block scalar
description: |
  Implements CEM and knossos changes with integration tests.
  Invoke for code changes, test writing, and build verification.

# Correct: quoted string
description: "Implements CEM and knossos changes with integration tests"

# Wrong: unquoted string with special characters
description: Implements CEM & knossos changes  # & is a YAML special character
```

### FlexibleStringSlice: Comma String vs. Array

Both formats are valid for `tools`:

```yaml
# Format 1: Comma-separated string
tools: Bash, Read, Glob

# Format 2: YAML array
tools:
  - Bash
  - Read
  - Glob

# Format 3: Inline YAML array
tools: [Bash, Read, Glob]
```

All three parse to the same `["Bash", "Read", "Glob"]` value. Array format is recommended for new agents because it avoids ambiguity with tool names that could contain commas.

### Validation Mode Differences

If an agent passes `ari agent validate` but fails `ari agent validate --strict`, the issue is a missing enhanced field:

| Strict-Only Requirement | Fix |
|------------------------|-----|
| `type is required in strict mode` | Add `type: specialist` (or appropriate type) |
| `tools is required` | Add `tools:` with at least one tool |
| `reviewer agents require contract.must_not` | Add `contract:` with `must_not:` entries |

### Unknown Tool Error

```
ERROR: tools: agent frontmatter: unknown tool "InvalidTool"
```

Valid built-in tools are: `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `Task`, `TodoWrite`, `TodoRead`, `WebSearch`, `WebFetch`, `Skill`, `NotebookEdit`, `AskUserQuestion`.

Tool names are case-sensitive. `bash` and `read` are not valid; use `Bash` and `Read`.

For MCP tools, use the `mcp:<server>` or `mcp:<server>/<method>` pattern.

### `ari agent update` Skips My Agent

The update command skips agents in three cases:

| Reason | Fix |
|--------|-----|
| `no type field` | Add `type:` to the frontmatter |
| `unknown archetype` | Use a valid type value (see [Type Assignment Guide](#type-assignment-guide)) |
| `no changes` | Platform and derived sections already match the current templates |

### Agent Has Extra Sections Not in Archetype

Extra sections (headings not defined in the archetype) are preserved at the end of the file during `ari agent update`. They are treated as author-owned and never removed. This is by design -- agents can have custom sections beyond the archetype template.

### Orchestrator Model Constraint

The schema requires orchestrators to use the `opus` model:

```
schema validation: orchestrator model must be opus
```

If an orchestrator needs a different model, consider whether `orchestrator` is the correct type. Orchestrators coordinate without executing work -- if the agent executes work, it may be a `specialist`.

## Compatibility Matrix

| Component | Version | Status |
|-----------|---------|--------|
| Agent frontmatter v1 (minimal: name, description, tools-as-string) | Pre-factory | Fully supported in WARN mode |
| Agent frontmatter v1.0 (enhanced: type, upstream, downstream, contract) | Factory | Fully supported in WARN and STRICT modes |
| `tools` as comma-separated string | Pre-factory | Supported via FlexibleStringSlice |
| `tools` as YAML array | Factory | Recommended format |
| Agents without `type` field | Pre-factory | Valid in WARN mode; skipped by `ari agent update` |
| MCP tool references (`mcp:server/tool`) | Factory | Warnings only for undeclared servers |
| Rite manifests without `mcp_servers` | Pre-factory | Fully compatible; no MCP servers materialized |
| Rite manifests with `mcp_servers` | Factory | Servers merged into `settings.local.json` |
