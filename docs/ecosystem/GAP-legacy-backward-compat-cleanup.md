### 2C. Spike's settings.json Cleanup Strategy Needs Refinement

The spike proposes adding `cleanupStaleBlanketSettings()` to `syncRiteScope()`. However:

1. `settings.json` is NOT generated by the sync pipeline today. No function in `internal/materialize/` writes or reads `settings.json`. The file was created by the now-deleted `writeDefaultSettings()` in `init.go`.
2. The sync pipeline generates `settings.local.json` via `config/hooks.yaml` -- this is the correct mechanism.
3. Adding cleanup to `syncRiteScope()` is safe but the detection logic must be precise. The spike says "if it contains only an `ari hook agent-guard` entry with no `--allow-path` flags." This is the right fingerprint. However, some satellites may have manually edited `settings.json` for legitimate reasons (adding custom hooks).
4. A safer approach: check provenance. If `settings.json` has NO provenance entry AND matches the exact blanket-deny fingerprint, delete it.

## 5. Provenance Integration Assessment

### Can Existing Provenance Handle Each Cleanup Item?

| Cleanup Item | Provenance Involvement | Assessment |
|---|---|---|
| `ARIADNE_*` env var rename | None needed | Pure code change, no file-level provenance |
| `LegacyDataDir()` / `MigrateLegacyPaths()` deletion | None needed | Dead code removal |
| Stale `settings.json` in satellites | **YES -- leverage provenance** | Files with NO provenance entry AND matching blanket-deny fingerprint are safe to delete. The orphan detection in `detectOrphansFromProvenance()` at `materialize.go:672` already handles "file not in provenance manifest" as orphan. However, current orphan detection is AGENTS-ONLY (scoped to `.claude/agents/` directory). Extending it to `settings.json` requires new code. |
| `.v2-backup` deletion | Indirect | Backup files are not tracked in provenance. Check is filesystem-only: if `*.v2-backup` files exist in `~/.claude/`, delete them. |
| `ariadne-msg-count-` prefix rename | None needed | Temp files in `os.TempDir()`, not in `.claude/` |

### Key Finding: Orphan Detection Cannot Handle settings.json Without New Code

The existing orphan detection pipeline (`detectOrphansFromProvenance` at `materialize.go:672-718`) operates exclusively on `.claude/agents/` entries. It:
1. Walks the `agents/` directory
2. Checks each file against the provenance manifest
3. Files not in manifest OR with `owner=user` are classified as orphans

To detect stale `settings.json`, the pipeline would need:
- A new function (e.g., `cleanupStaleBlanketSettings()`) that reads `.claude/settings.json`
- Content fingerprint matching (the exact JSON produced by the deleted `writeDefaultSettings()`)
- A provenance check: if `settings.json` has no provenance entry, it predates the manifest system

This is new code (~20-30 lines) in the materialize pipeline, not a configuration of existing orphan detection.

### Existing Provenance Capabilities That Help

1. **`LoadOrBootstrap()`** (`manifest.go:114`): Returns empty manifest if no file exists. Safe starting point for cleanup code.
2. **Divergence detection** (`divergence.go`): Can detect if a file's checksum differs from its provenance record. Not directly useful here (stale `settings.json` has no provenance record).
3. **Structural equality guard** (`manifest.go:86-110`): Prevents unnecessary writes. Cleanup code that modifies provenance must be aware of this guard.
