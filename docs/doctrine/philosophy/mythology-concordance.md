# Mythology Concordance

> The myth is the architecture. The architecture is the myth.

This concordance maps Greek mythology to the **SOURCE implementation** of Knossos—the canonical Go code, rite definitions, and platform source that **generates** the materialized `.claude/` projections.

**Critical Distinction:**
- **SOURCE** = Knossos repository (versioned, canonical, what Knossos IS)
- **PROJECTION** = `.claude/` directories (gitignored, materialized by `ari sync materialize`)

---

## Core Components

### Knossos (The Labyrinth)

**Mythological Origin:**
Knossos was the palace-complex on Crete, an elaborate labyrinth built to contain the Minotaur. The labyrinth's complexity protected and trapped in equal measure.

**Knossos Implementation:**
The Knossos repository itself—the platform SOURCE code, not the generated `.claude/` directories.

**Key Files:**
- Repository root (the labyrinth incarnate)
- `internal/` (Go packages implementing platform logic)
- `rites/` (canonical rite definitions—source of practice bundles)
- `cmd/ari/` (CLI source code)

**Design Rationale:**
Knossos is the SOURCE that creates projections. The labyrinth is the palace, not the rooms within it. Per ADR-0009 Amendment (2026-01-08): The Knossos repository IS the platform; `.claude/` directories are PROJECTIONS materialized into consuming projects.

---

### Ariadne (The Clew)

**Mythological Origin:**
Ariadne gave Theseus the clew (thread ball) that ensured his return from the labyrinth. The clew did not slay the Minotaur—it guaranteed safe return.

**Knossos Implementation:**
The `ari` CLI binary and its source code—the survival architecture ensuring deterministic return.

**Key Files:**
- `cmd/ari/` (CLI source, entry point: `main.go`)
- `internal/session/` (session management Go package)
- `internal/hook/clewcontract/` (clew contract types and event definitions)
- `internal/lock/` (concurrency safety mechanisms)

**Design Rationale:**
Ariadne is faithful, not intelligent. The CLI provides commands (`ari session`, `ari hook`, `ari sails`) that anchor, track, and cut the clew. Event recording (`events.jsonl`) is the physical manifestation of the thread.

---

### The Clew/Thread

**Mythological Origin:**
The physical thread that unwinds as you travel, providing a path back to the entrance.

**Knossos Implementation:**
Session state + event log (`events.jsonl`) + provenance trail.

**Key Files:**
- `internal/hook/clewcontract/` (event types, clew contract schema)
- `[session-dir]/events.jsonl` (materialized event log—PROJECTION)
- `[session-dir]/SESSION_CONTEXT.md` (session state—PROJECTION)

**Design Rationale:**
The clew is the only source of truth when context degrades. Events record decisions, tool calls, artifacts, errors. The Ship of Theseus problem: identity persists through transformation because the clew remembers what Theseus forgets.

### Theseus (The Navigator)

**Mythological Origin:**
Theseus entered the labyrinth to slay the Minotaur, using Ariadne's thread to ensure return.

**Knossos Implementation:**
The main Claude Code thread—the agentic intelligence that makes decisions, summons heroes, creates artifacts.

**Key Files:**
N/A (Theseus is the LLM agent itself, not a file)

**Design Rationale:**
Theseus has agency but amnesia. The clew compensates. Heroes (specialist agents) are summoned mid-journey via Task tool delegation, arriving with rich context from the clew.

### Heroes (Specialist Agents)

**Mythological Origin:**
Greek heroes summoned for specific labors (Heracles for strength, Odysseus for cunning).

**Knossos Implementation:**
Specialist agents invoked via Task tool, defined in rite manifests.

**Key Files:**
- `rites/[rite-name]/agents/` (rite-specific agent definitions)
- `agents/` (cross-cutting agents like Moirai)
- `.claude/agents/` (materialized projections—generated by `ari sync materialize`)

**Design Rationale:**
Heroes don't walk the whole labyrinth; they're summoned mid-journey. Better clew → better summoning context → better hero performance. Agents defined in SOURCE, materialized to PROJECTION.

### The Moirai (The Fates)

**Mythological Origin:**
Clotho spins the thread of life, Lachesis measures its allotment, Atropos cuts it when complete.

**Knossos Implementation:**
Centralized session lifecycle agent with event-driven activation.

**Key Files:**
- `agents/moirai.md` (agent definition—SOURCE, cross-cutting)
- `internal/session/` (Go implementation of lifecycle state machine)
- `.claude/agents/moirai.md` (materialized projection)

**Design Rationale:**
Only the Moirai may mutate session state. Clotho activates on `session_start`, Lachesis on state mutations, Atropos on `session_end`. Write guards block direct edits to `*_CONTEXT.md` files (ADR-0005, ADR-0013).

### The Inscription (CLAUDE.md)

**Mythological Origin:**
Words carved in stone at the labyrinth's entrance, telling travelers what to expect.

**Knossos Implementation:**
The `CLAUDE.md` file that declares available rites, agents, execution mode, and hooks.

**Key Files:**
- `knossos/templates/CLAUDE.md.tpl` (canonical template—SOURCE)
- `internal/inscription/` (Go package for Inscription generation)
- `.claude/CLAUDE.md` (materialized projection per project)

**Design Rationale:**
The labyrinth speaks at entry. SessionStart hooks read the Inscription to prepare Theseus. Knossos-managed sections (delimited by `<!-- KNOSSOS:START -->` ... `<!-- KNOSSOS:END -->`) are regenerated by `ari sync inscription`. Custom sections outside delimiters are preserved.

### White Sails (Confidence Signal)

**Mythological Origin:**
Aegeus watched from the cliff for his son's return. White sails = success, black sails = death. Theseus forgot to change the sails; Aegeus, seeing black, threw himself into the sea.

**Knossos Implementation:**
Computed confidence signal (WHITE/GRAY/BLACK) generated at session wrap.

**Key Files:**
- `internal/sails/` (Go package implementing confidence computation)
- `ari sails check` (CLI command)

**Design Rationale:**
Solves the Aegeus problem (false confidence). Three-color system prevents gaming. GRAY is honesty about uncertainty. Anti-gaming rules: no self-upgrade, proofs required for WHITE, QA review can elevate GRAY → WHITE.

### Rites (Practice Bundles)

**Mythological Origin:**
Ceremonies and invocations—structured practices for specific purposes.

**Knossos Implementation:**
Manifest-driven bundles of agents, skills, hooks, and workflows.

**Key Files:**
- `rites/` (canonical source definitions)
- `rites/[rite-name]/manifest.yaml` (rite composition definition)
- `internal/rite/` (Go package for rite loading and validation)
- `.claude/rites/` (materialized projections)

**Design Rationale:**
Rites are flexible compositions. Simple rites (skills-only) are valid. Invoke rites mid-journey (cheap) or swap rites (expensive full context switch). Rite source lives in `rites/`, projections materialize to `.claude/rites/` via `ari sync materialize`.

### Pythia (The Oracle)

**Mythological Origin:**
The Oracle at Delphi consulted before great journeys. Unlike the historical cryptic Pythia, Knossos's oracle speaks clearly.

**Knossos Implementation:**
Orchestrator agents that provide work breakdown, specialist routing, and checkpoint guidance.

**Key Files:**
- `rites/*/agents/pythia.md` (each rite's Pythia)

**Design Rationale:**
Consulted via `/consult` for workflow routing. Each rite has its own Pythia that decomposes initiatives into phases and routes to specialist heroes. Pythia agents carry Exousia contracts defining their jurisdictional boundaries. Clear guidance, not cryptic prophecy.

---

### Exousia (The Authority Contract)

**Mythological Origin:**
Exousia (exousia) in ancient Greek meant authority, jurisdiction, or the right to act. In the New Testament and classical texts, it distinguished between raw power (dynamis) and legitimate, bounded authority. Exousia was always contextual -- authority within a defined domain, not unlimited power.

**Knossos Implementation:**
The jurisdictional contract every agent carries, defining three boundaries: what the agent decides autonomously, what it escalates to Pythia or the user, and what it must never decide. Standardized as `## Exousia` with three required subsections: `### You Decide`, `### You Escalate`, `### You Do NOT Decide`.

**Key Files:**
- `## Exousia` section in every agent `.md` file across all rites

**Design Rationale:**
Agents without clear authority boundaries either under-act (escalating everything) or over-act (making decisions outside their domain). Exousia solves this by making jurisdiction explicit and auditable. The three-part structure -- Decide / Escalate / Do NOT Decide -- creates a complete authority surface that is both human-readable and machine-parseable.

### Naxos (Shore of Abandonment)

### Theoria (The Audit Delegation)

**Mythological Origin:**
Theoria was the official state delegation sent by a Greek polis to observe sacred festivals, consult oracles, or witness events at distant sanctuaries. The delegation traveled as a group, observed independently, and returned to report. The word shares its root with "theory": *theoria* was structured contemplation, knowledge gained through disciplined observation.

**Knossos Implementation:**
The `/theoria` audit operation—a composite primitive (dromena + legomena + agents) that dispatches parallel evaluators to assess domain health. The theoria is a shift from *praxis* to *theoria*: the hero pauses mid-journey to dispatch observers who survey the labyrinth itself.

**Key Files:**
- Dromena: `/theoria` (planned, forge rite)
- Legomena: `rites/shared/mena/pinakes/` (planned, domain registry)
- Agent: `rites/shared/agents/theoros.md` (planned, domain evaluator)

**Design Rationale:**
The theoria is structured observation, not intervention. Audit agents observe domains read-only, producing reports. The theoria uses the Argus Pattern for parallel dispatch. Findings surface Naxos items (abandoned work) and inform the Athens path (what to merge). The "State of the {X}" document becomes tribute for Minos, a map for Theseus, and evidence for White Sails.

### Theoroi (The Sacred Observers)

**Mythological Origin:**
Individual sacred observers within a theoria delegation. Chosen for judgment, dispatched to witness, bound to report honestly. A theoros was not a participant but a witness—valued for the clarity of seeing, not the strength of acting.

**Knossos Implementation:**
Domain evaluator agents—each dispatched via Task tool to assess a single domain using criteria from the Pinakes. Each theoros observes independently and produces a structured report.

**Key Files:**
- `rites/shared/agents/theoros.md` (planned, domain evaluator agent definition)

**Design Rationale:**
A theoros is not a hero. Heroes modify; theoroi observe. This distinction enforces read-only discipline in audit agents. Each theoros receives domain-specific criteria, evaluates independently, and returns a structured report. The naming prevents scope creep: if an agent is a theoros, it should not be making changes.

---

### Pinakes (The Domain Registry)

**Mythological Origin:**
Callimachus's Pinakes—the first systematic bibliography of the Library of Alexandria. Organized all knowledge by category, author, and work. Described what existed without containing it. The Pinakes was the map of the Library, not the Library itself.

**Knossos Implementation:**
The domain registry legomena—catalogs audit targets, evaluation criteria, grading rubrics, and report schemas. Persistent reference knowledge consulted by theoroi before and during observation.

**Key Files:**
- `rites/shared/mena/pinakes/` (planned, domain registry root)
- `rites/shared/mena/pinakes/INDEX.lego.md` (planned, registry index)
- `rites/shared/mena/pinakes/domains/` (planned, per-domain criteria files)
- `rites/shared/mena/pinakes/schemas/` (planned, report format schemas)

**Design Rationale:**
The Pinakes is a legomenon—persistent reference knowledge, progressively disclosed. It catalogs what domains exist and how to evaluate them, without containing the evaluation itself. Maps cleanly to the mena lifecycle model: persistent, loaded on-demand, consulted by theoroi. Always qualified as "the Pinakes" to distinguish from manifest files.

---

### Synkrisis (The Comparative Synthesis)

**Mythological Origin:**
Plutarch's synkrisis—the structured comparison that followed each pair of biographies in the *Parallel Lives*. Not summary but synthesis: truth through juxtaposition. After presenting two lives independently, Plutarch set them side by side to reveal patterns visible only in comparison.

**Knossos Implementation:**
The synthesis step following individual domain evaluations. Consolidates theoros reports into cross-domain patterns, systemic assessment, and the final "State of the {X}" attestation document.

**Key Files:**
- Planned: synthesis logic in `/theoria` dromena or dedicated synthesis agent

**Design Rationale:**
Individual domain reports reveal individual truths. Synkrisis reveals structural truths—compensatory patterns, systemic weaknesses, cross-domain dependencies. The synkrisis produces the artifact that Minos receives as tribute (the "State of" report). Seven theoroi return with seven truths; synkrisis reveals the eighth.

---

### Argus Pattern (N-Agent Parallel Observation)

**Mythological Origin:**
Argus Panoptes—the hundred-eyed giant set by Hera to watch over Io. His eyes never all closed at once; some slept while others watched. Total vigilance through distributed, overlapping observation. One body, a hundred eyes, nothing unseen.

**Knossos Implementation:**
The architectural pattern of launching N agents via Task tool in parallel, each observing a different domain. The main thread is the body; the agents are the eyes. Like "the Ship of Theseus" (identity through transformation) and "the Aegeus Problem" (false confidence), the Argus Pattern is a named pattern, not an entity.

**Key Files:**
- Pattern, not a file. Used by theoria; available for other parallel operations.

**Design Rationale:**
The Argus Pattern is a reusable technique, not a specific component. Its constraint—agents cannot spawn agents, only the main thread dispatches—maps to Argus's nature: one body, many eyes. The theoria is the first named user of this pattern; it is envisioned to serve parallel testing, parallel validation, tactical playbook swarms, and similar N-agent operations in future.

## Materialization Flow

**The Fundamental Truth:**

```
SOURCE (versioned, canonical)           PROJECTION (gitignored, generated)
rites/                               →  .claude/rites/
rites/*/agents/ + agents/            →  .claude/agents/
user-skills/                         →  .claude/skills/
internal/hook/ (Go) + rites/[rite]/hooks/ (scripts) → .claude/hooks/
knossos/templates/                   →  .claude/CLAUDE.md (rendered)
rites/[rite]/manifest.yaml           →  .claude/ACTIVE_RITE
```

**Command:** `ari sync materialize` reads SOURCE and writes PROJECTION.

Knossos (the SOURCE) generates `.claude/` (the PROJECTION). The labyrinth creates the rooms; the rooms are not the labyrinth.

## Mythology as Architecture

The mythology is not decoration—it's the design language. Each mythological element encodes:

1. **Architectural intent**: Why components exist
2. **Relationship patterns**: How components interact
3. **Constraint awareness**: What limits shape design
4. **Failure modes**: What happens when principles are violated

When you understand the myth, you understand the system. When you violate the myth, you violate the architecture.
