## Context

Knossos materializes files from multiple sources into `.claude/` directories. Every file in `.claude/` was either generated by Knossos (from rite definitions, templates, or mena sources) or created by the user (satellite agents, custom commands). Answering "who owns this file and where did it come from?" is essential for safe updates, orphan detection, and conflict resolution.

Today, three independent provenance systems answer this question using incompatible mechanisms:

### System 1: Source Resolver Types (`internal/materialize/source.go`)

The `SourceResolver` resolves rite locations via a 5-tier fallback chain and records the resolution in `ResolvedRite`:

```go
type SourceType string  // "project", "user", "knossos", "explicit", "embedded"

type ResolvedRite struct {
    Name         string     `json:"name"`
    Source       RiteSource `json:"source"`       // Type + Path + Description
    RitePath     string     `json:"rite_path"`
    ManifestPath string     `json:"manifest_path"`
    TemplatesDir string     `json:"templates_dir,omitempty"`
}
```

This tells you where a rite came from (project directory, user-level, Knossos platform, etc.) but operates at the rite level, not the individual file level. The result is written to `Result.Source` and `Result.SourcePath` as plain strings at `materialize.go:277-278`, then discarded after the materialization run. No persistent record remains.

### System 2: Usersync Manifests (`internal/usersync/manifest.go`)

The `usersync` package maintains per-resource-type JSON manifests (`USER_MENA_MANIFEST.json`, `USER_AGENT_MANIFEST.json`, etc.) at the user level:

```go
type Entry struct {
    Source      SourceType `json:"source"`       // "knossos", "knossos-diverged", "user"
    InstalledAt time.Time  `json:"installed_at"`
    Checksum    string     `json:"checksum"`     // SHA256 with "sha256:" prefix
    MenaType    string     `json:"mena_type,omitempty"`  // "dro" or "lego"
    Target      string     `json:"target,omitempty"`     // "commands" or "skills"
}
```

This tracks individual files with checksums and divergence detection (knossos vs. knossos-diverged vs. user). It is the most complete provenance implementation but operates only in the usersync pipeline (user-level `~/.claude/`), not in the project-level materialize pipeline.

### System 3: Inscription Manifest (`internal/inscription/types.go`)

The `KNOSSOS_MANIFEST.yaml` file tracks CLAUDE.md regions with ownership semantics:

```go
type Region struct {
    Owner              OwnerType  `yaml:"owner"`    // "knossos", "satellite", "regenerate"
    Source             string     `yaml:"source,omitempty"`
    PreserveOnConflict bool       `yaml:"preserve_on_conflict,omitempty"`
    Hash               string     `yaml:"hash,omitempty"`
    SyncedAt           *time.Time `yaml:"synced_at,omitempty"`
}
```

This has the richest ownership model (three owner types with conflict-resolution semantics) but applies only to CLAUDE.md regions, not to discrete files like agents, commands, or hooks.

### The Gap

Each resource type in `.claude/` uses a different detection method for ownership:

| Resource | Detection Method | Location | Weakness |
|----------|-----------------|----------|----------|
| Agents | Rite manifest membership | `materialize.go` (agent list check) | No record after materialization; user agent with same name as rite agent is overwritten |
| Rules | Template filename match | `materialize.go` (pattern matching) | User file with same name as template is overwritten |
| Hooks | Template filename match | `materialize.go` (pattern matching) | Same as rules |
| Mena (project) | None | `project_mena.go` | All projected mena are knossos-owned implicitly; no manifest |
| Mena (user) | Usersync manifest | `usersync/manifest.go` | Only for user-level sync |
| CLAUDE.md regions | KNOSSOS_MANIFEST.yaml | `inscription/types.go` | Only for region-level content |

No unified "who owns this file?" mechanism exists. Each pipeline phase reinvented detection because the need emerged incrementally. This makes orphan detection fragile (agents use manifest membership, everything else uses filename heuristics), prevents safe updates (no divergence detection for project-level files), and obscures debugging ("where did `.claude/commands/foo/` come from?").

## Decision

Introduce a unified provenance record that tracks every file Knossos places in `.claude/`, regardless of resource type or pipeline. The design extends the existing `usersync` manifest schema -- already the most complete implementation -- to cover all files touched by the materialize pipeline.

### Provenance Record Schema

A single manifest (extending the existing `KNOSSOS_MANIFEST.yaml` or stored alongside it) tracks per-file provenance:

```yaml
schema_version: "3.0"
last_sync: "2026-02-08T12:00:00Z"
active_rite: hygiene
files:
  agents/orchestrator.md:
    owner: knossos            # knossos | satellite | user
    source_pipeline: materialize
    source_path: rites/hygiene/agents/orchestrator.md
    source_type: project      # project | user | knossos | explicit | embedded
    checksum: "sha256:abc123..."
    installed_at: "2026-02-08T12:00:00Z"
  agents/my-custom-agent.md:
    owner: user
    source_pipeline: null
    source_path: null
    source_type: null
    checksum: "sha256:def456..."
    installed_at: "2026-02-07T10:00:00Z"
  commands/commit/:
    owner: knossos
    source_pipeline: materialize
    source_path: mena/operations/commit/INDEX.dro.md
    source_type: project
    checksum: "sha256:ghi789..."
    installed_at: "2026-02-08T12:00:00Z"
  rules/internal-hook.md:
    owner: knossos
    source_pipeline: materialize
    source_path: knossos/templates/rules/internal-hook.md
    source_type: project
    checksum: "sha256:jkl012..."
    installed_at: "2026-02-08T12:00:00Z"
  hooks/hooks.yaml:
    owner: knossos
    source_pipeline: materialize
    source_path: knossos/templates/hooks/hooks.yaml
    source_type: project
    checksum: "sha256:mno345..."
    installed_at: "2026-02-08T12:00:00Z"
```

### Go Types

```go
package materialize

// FileOwner represents who owns a file in .claude/.
type FileOwner string

const (
    OwnerKnossos   FileOwner = "knossos"   // Generated by Knossos (rite, template, mena)
    OwnerSatellite FileOwner = "satellite" // Belongs to satellite project (user-modified knossos file)
    OwnerUser      FileOwner = "user"      // Created by user, not from any knossos source
)

// ProvenanceEntry tracks the origin and state of a single file in .claude/.
type ProvenanceEntry struct {
    Owner          FileOwner  `yaml:"owner"                    json:"owner"`
    SourcePipeline string     `yaml:"source_pipeline,omitempty" json:"source_pipeline,omitempty"` // "materialize" or "usersync"
    SourcePath     string     `yaml:"source_path,omitempty"    json:"source_path,omitempty"`      // Relative path to source file
    SourceType     string     `yaml:"source_type,omitempty"    json:"source_type,omitempty"`      // SourceType from source resolver
    Checksum       string     `yaml:"checksum"                 json:"checksum"`                   // SHA256 with "sha256:" prefix
    InstalledAt    time.Time  `yaml:"installed_at"             json:"installed_at"`
}

// ProvenanceManifest is the unified file-level provenance tracker for .claude/.
type ProvenanceManifest struct {
    SchemaVersion string                       `yaml:"schema_version" json:"schema_version"`
    LastSync      time.Time                    `yaml:"last_sync"      json:"last_sync"`
    ActiveRite    string                       `yaml:"active_rite"    json:"active_rite"`
    Files         map[string]*ProvenanceEntry  `yaml:"files"          json:"files"` // Key: relative path within .claude/
}
```

### Key Properties

1. **Unified**: One mechanism for agents, mena, rules, hooks, and settings. Replaces per-resource-type heuristics.
2. **Source tracking**: Every knossos-owned file records its source path (the file it was copied from) and source type (which tier of the resolution chain it came from).
3. **Checksum-based divergence**: If a knossos-owned file is modified locally, the checksum mismatch at next sync signals "diverged" status. This reuses the pattern already proven in `internal/usersync/`.
4. **User files are sacred**: Files with `owner: user` are never overwritten. Files not in the manifest are assumed to be user-created and are left untouched.
5. **Pipeline-aware**: The `source_pipeline` field distinguishes materialize-placed files from usersync-placed files, enabling each pipeline to manage its own files without interfering with the other.

### Ownership Semantics

| Owner | On Sync | On Conflict | On Orphan Detection |
|-------|---------|-------------|---------------------|
| `knossos` | Overwrite with latest source content | Source always wins | File removed when no longer in rite manifest |
| `satellite` | Preserve; warn if source changed | User version preserved; diff available | Preserved (never removed automatically) |
| `user` | Skip entirely | N/A (no knossos source to conflict with) | Preserved (never removed automatically) |

Divergence detection: on sync, if a file with `owner: knossos` has a checksum that does not match the on-disk content, it is promoted to `owner: satellite`. This mirrors the `knossos-diverged` state from `usersync.SourceType`.

### Query Operations

The manifest enables direct answers to operational questions:

- **"What files did rite X place?"**: Filter by `active_rite` at install time.
- **"Is this file safe to overwrite?"**: Check `owner` field.
- **"What changed since last sync?"**: Compare on-disk checksums to manifest checksums.
- **"What files are orphaned?"**: Files in manifest not present in current rite's agent/mena/hook lists.
- **"Where did this file come from?"**: Read `source_path` and `source_type`.

### Relationship to Existing Manifests

| Existing Manifest | Disposition |
|-------------------|-------------|
| `KNOSSOS_MANIFEST.yaml` (inscription) | **Retained**. Continues to track CLAUDE.md region-level provenance (owner, hash, synced_at). Region-level and file-level are complementary, not redundant. |
| `USER_*_MANIFEST.json` (usersync) | **Retained for user-level**. The usersync manifests track `~/.claude/` provenance. The new provenance manifest tracks project-level `.claude/` provenance. They serve different scopes. |
| `Result.Source` / `Result.SourcePath` | **Superseded**. The provenance manifest provides persistent, per-file source tracking that replaces these ephemeral per-run strings. The `Result` fields may be retained for backward compatibility but become redundant. |

### Storage Location

`{project}/.knossos/PROVENANCE_MANIFEST.yaml` -- stored inside the `.claude/` directory it describes, alongside `KNOSSOS_MANIFEST.yaml`. This keeps provenance data co-located with the files it tracks and ensures it is available to any tool reading the `.claude/` directory.

The manifest itself is owned by Knossos and is always overwritten on sync (it is a derived artifact, not user content).

## Migration Path

### Phase 1 (current sprint -- WU-014)

Remove the unused `MenaScope` infrastructure. This eliminates the mena-specific provenance attempt that was never adopted. Clears the way for the unified approach.

### Phase 2 (future sprint)

Implement `ProvenanceManifest` in the materialize pipeline:

1. Add `ProvenanceManifest` types to `internal/materialize/`.
2. During `MaterializeWithOptions`, after each file write (agents, mena, rules, hooks), record a `ProvenanceEntry` with source path, source type, and checksum.
3. Write `PROVENANCE_MANIFEST.yaml` at the end of materialization.
4. On subsequent materializations, load existing manifest to detect diverged files before overwriting.

### Phase 3 (future sprint)

Leverage the manifest for improved operations:

1. Replace agent orphan detection (currently based on rite manifest membership) with manifest-based detection.
2. Add `ari provenance show` CLI command for debugging.
3. Implement divergence warnings: "File X was modified locally; use --force to overwrite."

### Phase 4a (complete -- schema unification)

Unified usersync and provenance manifests under a single schema. 14 stakeholder decisions confirmed:

1. Full schema unification — one `ProvenanceEntry` for both scopes
2. Owner model: `knossos`/`user`/`untracked`. Divergence computed, not stored.
3. `ScopeType` replaces `SourcePipeline`. Values: `rite`/`user`.
4. Schema version bumped to `2.0` with `migrateV1ToV2()` for backward compat.
5. Single user manifest: `USER_PROVENANCE_MANIFEST.yaml` replaces 3 JSON manifests.
6. Manifest keys namespaced: `agents/`, `commands/`, `skills/`, `hooks/`.
7. `CollisionChecker` reads rite provenance manifest (fallback: directory scan).
8. Orphan parity: user-level auto-removes knossos-owned orphans.
9. CLI: `ari provenance show` gains SCOPE column and `--scope` filter.
10. Clean break migration: old JSON manifests backed up as `.v2-backup`.

### Phase 4b (complete -- pipeline absorption)

Absorbed usersync into materialize. 26 stakeholder decisions confirmed:

1. Full code merge — `internal/usersync/` deleted, sync logic in `internal/materialize/user_scope.go`.
2. Scope-gated pipeline — stages conditionally execute per scope (rite/user/all).
3. Unified CLI: `ari sync [--scope] [--rite] [--source] [--resource] [--dry-run] [--recover] [--overwrite-diverged] [--keep-orphans]`.
4. Default scope=all. `ari sync` syncs both rite and user scopes.
5. Rite first, then user (collision detection needs fresh rite manifest).
6. `--force` removed. Sync always idempotent. `--overwrite-diverged` for diverged files.
7. `--minimal` eliminated. `ari init` or scope gating handles cross-cutting mode.
8. Orphans: auto-remove knossos-owned by default. `--keep-orphans` override.
9. Promotion dropped. Users put global agents in source dirs.
10. Remote sync commands removed (status/pull/push/diff/resolve/history/reset).
11. CollisionChecker moved to materialize (manifest-based, no directory scan fallback).
12. SourceResolver extended for user-scope resolution.
13. Smart fallback: scope=all skips rite silently if no ACTIVE_RITE; scope=rite errors.
14. `internal/sync/` package pruned (remote sync methods deleted, state.go kept for trackState).
15. 60+ mena/rites content files updated to reference new CLI.

5 sprints: types+skeleton → user pipeline → CLI rebuild → dead code+tests → content migration.

## Consequences

### Positive

1. **Single source of truth for file ownership.** Every file in `.claude/` has a recorded owner, source, and checksum. No more per-resource-type heuristics.
2. **Safe updates without data loss.** Divergence detection prevents overwriting user modifications without warning. The `satellite` owner state preserves local changes.
3. **Reliable orphan detection.** Files present in the manifest but absent from the current rite's resource lists are orphans. No filename heuristics needed.
4. **Debuggable materialization.** "Where did this file come from?" is answered by reading one manifest entry instead of tracing through materialization code.
5. **Foundation for multi-rite coexistence.** When multiple rites need to coexist, the manifest tracks which rite placed each file, enabling clean rite switching.
6. **Reuses proven patterns.** The checksum-based divergence detection and three-state ownership model are already working in `internal/usersync/`. This extends the pattern, not invents a new one.

### Negative

1. **Additional I/O per materialization.** Reading and writing the provenance manifest adds one file read and one file write per sync. For the current file count (~80 files in `.claude/`), the manifest is small (< 10KB). Impact is negligible.
2. **Schema evolution burden.** A new manifest format requires versioning and migration. Mitigated: the schema starts at version 3.0 and uses the same backup-on-version-mismatch strategy as the usersync manifest (`manifest.go:77-84`).
3. **Two manifests in `.claude/`.** `KNOSSOS_MANIFEST.yaml` (regions) and `PROVENANCE_MANIFEST.yaml` (files) serve different granularities. This could confuse contributors. Mitigated: clear naming and documentation. The region manifest may eventually be absorbed into the provenance manifest if CLAUDE.md generation becomes file-level rather than region-level.
4. **Retroactive adoption.** Existing `.claude/` directories have no provenance manifest. The first sync after adoption must either: (a) treat all existing files as knossos-owned (risks overwriting user files), or (b) treat unknown files as user-owned (safe default, may miss orphans on first run). Option (b) is recommended.

### Neutral

1. **No behavioral change to existing pipelines.** The provenance manifest is additive. Materialization, usersync, and inscription continue to work as before. The manifest is written alongside existing operations, not instead of them.
2. **Inscription manifest retained.** CLAUDE.md region-level tracking is complementary to file-level tracking. The two manifests serve different purposes and coexist without conflict.

## Alternatives Considered

### Alternative 1: Extend KNOSSOS_MANIFEST.yaml with a `files:` Section

Add file-level provenance entries directly to the existing inscription manifest rather than creating a new file.

**Rejected**: KNOSSOS_MANIFEST.yaml is tightly coupled to the inscription pipeline's region model. Adding file-level entries conflates two concerns (region ownership and file ownership) in one schema, complicating both the inscription pipeline and the new provenance system. Separate files with clear responsibilities are simpler to maintain.

### Alternative 2: Embed Provenance in File Headers

Add provenance metadata as comments or frontmatter at the top of each file in `.claude/` (e.g., `<!-- knossos:owner=knossos source=rites/hygiene/agents/orchestrator.md -->`).

**Rejected**: This modifies file content, which breaks checksum-based idempotency (the file's content includes its own metadata). It also does not work for non-text files (YAML settings, JSON configs). A sidecar manifest is cleaner.

### Alternative 3: Use Git Attributes or Ignore Patterns

Track ownership via `.gitattributes` or custom metadata in the git object store.

**Rejected**: Knossos must work in repositories without git and in scenarios where `.claude/` is gitignored. Git-based provenance creates a hard dependency on git and fails for the common case of `.claude/` being excluded from version control.

### Alternative 4: Unify Usersync and Project Manifests Immediately

Build one manifest package that handles both `~/.claude/` (user-level) and `{project}/.claude/` (project-level) provenance from day one.

**Rejected for now**: The two pipelines have fundamentally different semantics. Usersync is additive (never removes user files, collision detection, recovery mode). Materialize is authoritative (rite defines the complete set, orphans are detected). Forcing both into one abstraction prematurely would create a leaky abstraction. Phase 4 of the migration path revisits unification if the schemas converge naturally.

## Implementation

The unified provenance model has been implemented across the materialization pipeline. Key implementation details:

### Package Structure

- **Package**: `internal/provenance/`
- **Core files**:
  - `provenance.go` - Core types (ProvenanceManifest, ProvenanceEntry, OwnerType)
  - `manifest.go` - Manifest loading, saving, and validation
  - `collector.go` - Provenance entry collection during materialization
  - `divergence.go` - Divergence detection and ownership promotion

### Pipeline Integration

The provenance collector is wired into all 9 materialize stages:

1. **CLAUDE.md inscription** - Records inscription-generated CLAUDE.md regions
2. **Agent materialization** - Tracks agent files from rite manifests
3. **Rules materialization** - Records rule files from templates
4. **Hooks materialization** - Tracks hooks.yaml and hook scripts
5. **Settings merge** - Records settings.json ownership
6. **MCP server merge** - Tracks MCP server configurations
7. **Workflow materialization** - Records workflow.md if present
8. **Project mena projection** - Tracks projected commands and skills (directory-level)
9. **Shared mena projection** - Tracks cross-rite shared mena

Each stage calls `collector.Record()` after writing files, capturing:
- Relative path within `.claude/`
- Owner type (knossos, user, unknown)
- Source path (where the file came from)
- Source type (project, user, knossos, etc.)
- SHA256 checksum with "sha256:" prefix
- Timestamp

### CLI Command

**Command**: `ari provenance show`

Displays a table of all files in `.claude/` with their ownership and divergence status:

```
PATH                          OWNER     SOURCE                          STATUS
agents/orchestrator.md        knossos   rites/ecosystem/agents/...      match
agents/my-custom.md           user      (user-created)                  -
commands/commit/              knossos   mena/commit                     match
rules/internal-hook.md        knossos   templates/rules/internal-...    diverged
```

Supports `--output json` for structured output and `--verbose` for full checksums.

### Orphan Detection Refactor

The `detectOrphans()` function in `internal/materialize/materialize.go` now uses manifest-based detection when a provenance manifest exists:

- **With manifest**: An agent is an orphan if:
  - Not in provenance manifest, OR
  - Has `owner: user` in manifest, OR
  - Has `owner: knossos` but not in current rite's agent list
- **Without manifest**: Falls back to legacy rite manifest membership check (backward compatible)

This replaces the previous rite-manifest-only detection, enabling accurate orphan detection across rite switches and user customizations.

### Key Decisions

Ten implementation decisions were made during stakeholder interview (documented in Context Design):

1. **Owner type naming**: Use "knossos/user/unknown" (not "knossos/satellite/user") for simplicity
2. **Manifest filename**: `PROVENANCE_MANIFEST.yaml` (not extending KNOSSOS_MANIFEST.yaml)
3. **Unknown promotion**: Treat unknown as user-owned (safe default)
4. **Mena directory tracking**: Track directories (not individual files) for mena commands
5. **Divergence on deletion**: User-deleted files are promoted to user ownership
6. **Bootstrap on missing manifest**: LoadOrBootstrap returns empty manifest (no error)
7. **Checksum format**: Use "sha256:" prefix per ADR-0026 and internal/checksum convention
8. **CLI status column**: Show "match/diverged/-/missing" for clarity
9. **JSON output format**: Return raw manifest for structured consumption
10. **User file source display**: Show "(user-created)" instead of empty fields

- All existing tests pass (including integration tests)
- Provenance collector tested with 9-stage materialization
- Divergence detection validated with ownership promotion scenarios
- Orphan detection tested with rite switching and user agent preservation
