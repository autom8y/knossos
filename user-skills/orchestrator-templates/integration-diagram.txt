DEVELOPER WORKFLOW
──────────────────────────────────────────────────────────────────────────────

   Developer Creates Team
          │
          ├─→ [1] Create orchestrator.yaml
          │        team metadata, routing, handoff criteria, skills
          │        Location: teams/{team}/orchestrator.yaml
          │        Format: YAML (human-readable config)
          │
          ├─→ [2] Run Generator
          │        /roster/templates/orchestrator-generate.sh
          │        Input: orchestrator.yaml + workflow.yaml
          │        Output: orchestrator.md (production ready)
          │
          ├─→ [3] Validate Output
          │        /roster/templates/validate-orchestrator.sh
          │        Checks: 10 structural and semantic rules
          │        Exit: 0 (success) or 2-3 (failure)
          │
          ├─→ [4] Review Generated File
          │        orchestrator.md looks correct?
          │        Specialists and routing preserved?
          │
          ├─→ [5] Commit Both Files
          │        git add orchestrator.yaml orchestrator.md
          │        Both files committed to maintain consistency
          │
          └─→ [6] Test Team Activation
                   ./swap-team.sh my-team
                   Frontmatter parses?
                   Team active?

INTEGRATION POINTS
──────────────────────────────────────────────────────────────────────────────

orchestrator.md ─→ swap-team.sh
                   Reads frontmatter
                   Validates format
                   Parses: name, role, description, model, color, tools
                   └─→ Activates team (symlinks to active agent position)

orchestrator.md ─→ CEM Sync
                   Distributes across satellites
                   Validates: markdown syntax, YAML frontmatter
                   Tracks in AGENT_MANIFEST
                   └─→ Makes orchestrator available on all satellites

orchestrator.md ─→ AGENT_MANIFEST
                   Discovered by agent scanning
                   Registered as "orchestrator" agent
                   Source: "team", Origin: team name
                   └─→ Agent discovery and team activation

workflow.yaml ─→ orchestrator-generate.sh
                   Generator validates specialist names
                   Source of specialist definitions
                   └─→ Ensures routing table references valid agents

orchestrator.yaml ← schema validation
                   Checked against JSON schema
                   Ensures required fields present
                   Validates specialist count (3-6)
                   └─→ Prevents invalid configurations


TEAM PERSPECTIVE
──────────────────────────────────────────────────────────────────────────────

Day 1: Create New Team
   ├─→ Create orchestrator.yaml (copy template, edit)
   ├─→ Run orchestrator-generate.sh my-team
   ├─→ Run validator
   ├─→ git add + git commit
   ├─→ ./swap-team.sh my-team
   └─→ Test works!

Later: Update Team's Routing
   ├─→ Edit orchestrator.yaml (add specialist, change routing)
   ├─→ Run orchestrator-generate.sh my-team
   ├─→ Run validator
   ├─→ git add + git commit
   └─→ New routing takes effect

Later: Update All Teams (New Protocol)
   ├─→ Edit orchestrator-base.md.tpl (add section, improve wording)
   ├─→ for team in all_teams; orchestrator-generate.sh $team; done
   ├─→ for team in all_teams; validate $team; done
   ├─→ Review diffs (all should be cosmetic)
   ├─→ git add + git commit (all orchestrators)
   └─→ All teams get improvement automatically

KEY PRINCIPLES
──────────────────────────────────────────────────────────────────────────────

1. YAML is Source of Truth
   └─→ orchestrator.yaml contains team's design decisions
       orchestrator.md is generated (can be regenerated anytime)

2. Template is Shared Protocol
   └─→ All orchestrators use same template
       Changes to protocol benefit all teams at once

3. Generation is Deterministic
   └─→ Same YAML + Same template = Same .md file
       Easy to validate and test

4. Validation Happens at Two Levels
   └─→ Schema validation (YAML structure)
       Post-generation validation (markdown quality)

5. No Manual Edits to Generated Files
   └─→ Edit YAML, not .md
       Regenerate when ready
       Durable abstraction: YAML survives implementation changes
