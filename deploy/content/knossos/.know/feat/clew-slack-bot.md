# Clew Organizational Intelligence Slack Bot

## Purpose and Design Rationale

Clew is a Slack-native organizational intelligence assistant that answers questions about engineering codebases using pre-indexed `.know/` domain knowledge. It surfaces structured knowledge (architecture, conventions, scar-tissue, test-coverage, design-constraints) generated by the knossos theoria pipeline. Design commitments: honest answers with stated confidence levels (HIGH/MEDIUM/LOW tiers), pre-baked content over live indexing, fail-open architecture throughout, HTTP webhook mode (not Socket Mode), separation of triage and reasoning packages to avoid circular dependencies, and three-tier streaming response degradation.

## Conceptual Model

**Knowledge address space:** Qualified names `org::repo::domain`. **Trust tier model:** geometric mean of freshness x retrieval x coverage -> HIGH/MEDIUM/LOW layouts. **Query lifecycle:** event dedup (5-min TTL) -> concurrency gate (max 10) -> conversation history -> triage (4-stage RAG) -> reasoning pipeline -> progressive render. **Conversation memory:** hybrid window (last N verbatim + Haiku summary), 4-state FSM (CREATED->ACTIVE->DORMANT->RESURRECTING). **Content delivery:** `contentLookup()` closure populates SearchEntry.Description with real `.know/` content (Sprint 4-prime fix).

## Implementation Map

Entry: `ari serve` (`internal/cmd/serve/serve.go`). HTTP surface: `/slack/events` (HMAC-SHA256 verified), `/health`, `/ready`. Key packages: `internal/slack` (handler, client, renderer, streaming, conversation), `internal/reason` (pipeline, intent, context, response), `internal/triage` (4-stage RAG), `internal/trust` (confidence scoring), `internal/search/knowledge` (semantic index), `internal/observe` (CloudWatch EMF + OTEL), `internal/envload` (4-tier config), `internal/serve` (HTTP infrastructure). Deployment: distroless Docker on ECS Fargate (256 CPU/512MB), ALB + HTTPS, Secrets Manager for API keys.

Layer invariants: reason/ does NOT import serve/ or cmd/; trust/ and search/ do NOT import reason/; reason/ does NOT import triage/ (data transfer via mirrored structs, slack/ bridges).

## Boundaries and Failure Modes

Fail-open throughout: missing API key -> pipeline nil, server starts; LLM unavailable -> BM25 fallback; KnowledgeIndex build timeout -> BM25 fallback; Slack retry -> dedup drops; burst -> rate limit message; pipeline timeout (60s) -> error status; triage failure -> v1 fallback. Bot message filtering prevents prompt injection. Single ECS task (no redundancy). `SlackThreadFetcher` nil at runtime (future PR). Stale `.know/` content between image rebuilds shows MEDIUM confidence warning.
