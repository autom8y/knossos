## Error Handling Style

### Custom Exception Classes

The project uses a custom exception class per integration boundary, not plain `ValueError` or bare `Exception` raises:

- `src/autom8_sms/clients/twilio.py` -- `TwilioError(Exception)` with `__init__(self, message: str, code: int | None = None)`. Stores the Twilio numeric error code on `self.code` for downstream discrimination (e.g., opt-out code `21610` checked in `reminder/orchestrator.py` at line 188).

No other custom exceptions are defined in this codebase. The project relies on platform-provided exception types (`DataServiceUnavailableError`, `DataServiceError`) imported from `autom8y_interop.data`.

### Error Propagation

**Catch-at-boundary pattern**: Both Lambda handlers use a top-level `try/except Exception as e` that converts to `{"statusCode": 500, "body": {"error": str(e)}}`. Internal service methods let exceptions propagate upward to this boundary.

The `ConversationOrchestrator.process_conversation()` method (line 278) wraps the entire flow in `except Exception as e` and returns a `ProcessingResult(success=False, error=str(e))` -- a result-type pattern that avoids raising through the outer caller.

**Error wrapping convention**: `raise ... from ...` is not used. When re-raising or converting errors, the pattern is `str(e)` to capture message text only -- no chaining.

**Fault isolation loop**: In `ReminderOrchestrator.run()`, each appointment is processed in a `try/except TwilioError` (specific) followed by `except Exception` (catch-all), with `continue` so one failure never aborts the loop. This serial fault-isolation is the canonical pattern for batch Lambda work.

**Three-layer exception handling in mark_reminder_sent** (lines 165-179 of `reminder/orchestrator.py`): Inner `try/except Exception` around a non-critical sub-operation, with `logger.error()` but no re-raise -- "log and swallow" for non-fatal continuations.

### Error Handling at Lambda Boundary

```python
@instrument_lambda
def lambda_handler(event: dict[str, Any], context: Any) -> dict[str, Any]:
    try:
        result = asyncio.run(...)
        return {"statusCode": 200, "body": result}
    except Exception as e:
        logger.exception("handler_failed", error=str(e))
        return {"statusCode": 500, "body": {"error": str(e)}}
```

`logger.exception()` is used (not `logger.error()`) at the Lambda boundary -- captures full traceback in structured logs.

### Validation Error Pattern

Server-side validation (SD-03, SD-04, SD-05 in `tools/validation.py`) returns a `ValidationResult` dataclass with `valid: bool` and `error_message: str | None`. No exceptions are raised for invalid input from Claude -- callers check `result.valid` and branch accordingly. This is the "result type" pattern for AI-generated input guard paths.

### Contract Validation

`ValidationError` from Pydantic is caught at HTTP response boundaries (e.g., `data_service.py` line 291): logs `logger.error("contract_validation_failed", ...)` and returns `None` -- no raise. This avoids contract failures from an external service crashing the entire conversation.

## Domain-Specific Idioms

### `from_env()` Classmethod

Every client and orchestrator exposes a `from_env()` classmethod that wires up all dependencies from environment config. This is the canonical construction path in Lambda cold-start context:

```python
@classmethod
def from_env(cls) -> TwilioClient:
    """Create client from environment variables."""
    return cls(TwilioConfig())
```

Used in: `TwilioClient`, `SmsDataClient`, `ConversationOrchestrator`, `ReminderOrchestrator`.

### Async Context Manager on Clients

All clients implement `__aenter__` / `__aexit__` for resource cleanup (HTTP connection close). The Lambda handler uses `async with Client.from_env() as client:` as the idiomatic usage:

```python
async with SmsDataClient.from_env() as data_client:
    ...
async with ConversationOrchestrator.from_env() as orchestrator:
    ...
async with ReminderOrchestrator.from_env() as orchestrator:
    ...
```

`close()` (sync) and `close_async()` (async) coexist on `SmsDataClient` for callers that cannot use context managers.

### Dataclass vs BaseModel Split

- **`@dataclass`**: Used for internal result objects that carry computation results between layers -- `ProcessingResult`, `GenerationResult`, `ToolDispatchResult`, `ValidationResult`. Mutable dataclasses for results, `frozen=True` for value-type results.
- **`BaseModel`**: Used for all API contract models (request/response shapes), settings classes, and formatted output shapes passed to Claude. `ConfigDict(frozen=True)` on purely data-holding models like `FormattedDaySlots`.

Rule of thumb: BaseModel = wire/storage contract; dataclass = internal computation result.

### `from __future__ import annotations`

Present in every file that has type annotations involving forward references or `TYPE_CHECKING` guards. Used universally across the codebase.

### `TYPE_CHECKING` Guard

Protocol types and heavy interop imports are gated behind `if TYPE_CHECKING:` to avoid circular imports and reduce Lambda cold-start time:

```python
if TYPE_CHECKING:
    from autom8y_ai import AIClientProtocol
    from autom8y_interop.data.protocols import DataMessageProtocol
```

### Structured Logging Convention

Every module creates a logger via `logger = get_logger(__name__)` (from `autom8y_log`). Log messages use **snake_case event names as the first positional argument**, followed by keyword arguments as structured fields:

```python
logger.info("reminder_sent", appointment_id=appointment_id, office_phone=..., lead_phone=...)
logger.warning("pilot_phone_malformed", phone=phone)
logger.exception("handler_failed", error=str(e))
```

No f-strings in log messages -- all variable context goes in kwargs.

### Phone Number Masking in Logs

Phone numbers are always partially masked in log output: `lead_phone[:4] + "****"`. Only the first 4 digits of E.164 numbers are logged.

### OTEL Tracing with Named Tracers

Two tracers are declared at module level in `orchestrator.py`:
```python
_ai_tracer = otel_trace.get_tracer("autom8y.gen_ai")
_sms_tracer = otel_trace.get_tracer("autom8y.sms")
```
`@instrument_lambda` decorator from `autom8y_telemetry.aws` wraps Lambda handlers. `@trace_validation(...)` decorator from `autom8y_telemetry` wraps individual validation functions.

### Reference Tags in Comments

Comments frequently reference requirement IDs (REQ-N), security decisions (SD-04, SD-05), work streams (WS-3, WS-D), ADRs (ADR-002, ADR-007, ADR-008), and edge-case codes (EC-7, EC-11, EC-13). These serve as cross-references to external design documents.

### Tool Schemas as Module-Level Dicts

Claude tool_use JSON schemas are defined as `dict[str, Any]` constants at module level in `tools/definitions.py`, then composed into list constants (`SCHEDULING_TOOLS`, `CANCEL_TOOLS`, `RESCHEDULE_TOOLS`). No classes -- pure data.

### Security-by-Design: Server-Side Resolution

Phone numbers (`office_phone`, `lead_phone`) and idempotency keys are **never** accepted from Claude's tool inputs. They are always resolved server-side from `ConversationContext` or generated via `uuid.uuid4()`. This is documented as pattern `SD-03`. Code enforces this with `assert` preconditions and explicit comments at every tool handler.

### Dead-Man's Switch Metric Pattern

The appointment reminder Lambda emits a success timestamp metric via `emit_success_timestamp(NAMESPACE)` from `autom8y_telemetry.aws`. This is a CloudWatch dead-man's switch: if the Lambda stops succeeding, the timestamp goes stale and triggers an alarm. The metric namespace is module-level constant `NAMESPACE = "Autom8y/SmsReminder"`.
