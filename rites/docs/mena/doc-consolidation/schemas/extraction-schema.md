# Extraction Schema

> **Purpose**: Define the structure for `extraction-{topic}.yaml` files that capture analyzed source content, enabling synthesis agents to work from structured extractions (~2K tokens) rather than raw sources (~10K tokens).

## Design Principles

1. **Semantic Capture**: Store understanding, not just references
2. **Conflict Visibility**: Surface disagreements explicitly for resolution
3. **Authority Ranking**: Identify which source is canonical for each concept
4. **Token Efficiency**: Structured data over prose; references over duplication

---

## Schema Definition

```yaml
# extraction-{topic}.yaml
# Generated by: Extractor Agent
# Consumed by: Synthesis Agent

# Required: Schema version for forward compatibility
schema_version: "1.0"

# Required: What this extraction covers
topic:
  id: string           # Kebab-case identifier (e.g., "settings-merge")
  name: string         # Human-readable name (e.g., "Settings Merge Algorithm")
  scope: string        # Brief description of extraction boundaries

# Required: When this extraction was created/updated
metadata:
  created_at: string   # ISO 8601 timestamp
  updated_at: string   # ISO 8601 timestamp
  extractor: string    # Agent/process that created this (e.g., "extractor-agent-v1")
  source_token_count: integer  # Approximate tokens in source files
  extraction_token_count: integer  # Tokens in this extraction (target: <2K)

# Required: Source files analyzed
sources:
  - path: string       # Relative path from repo root
    hash: string       # SHA-256 of file content (for staleness detection)
    authority: enum    # "primary" | "secondary" | "supplementary"
    sections_used:     # Which parts of the file were relevant
      - heading: string
        line_range: [integer, integer]  # [start, end] inclusive

# Required: Canonical sections extracted
canonical_sections:
  - id: string         # Section identifier (e.g., "merge-algorithm")
    heading: string    # Section title for output document
    authority_source: string  # Path of authoritative source for this section
    content_summary: string   # 2-3 sentence summary of what this section covers
    key_points:        # Bullet points capturing essential information
      - string
    code_examples:     # Optional: relevant code snippets
      - language: string
        snippet: string
        source: string   # File:line reference
    cross_references:  # Links to other sections/topics
      - target: string   # Section ID or topic ID
        relationship: enum  # "depends_on" | "extends" | "supersedes" | "related"

# Optional: Concepts that appear across multiple sources
shared_concepts:
  - concept: string    # Concept name
    definition: string # Canonical definition
    sources:           # Where this concept appears
      - path: string
        treatment: string  # How this source describes/uses the concept

# Required: Conflicts detected between sources
conflicts:
  - id: string         # Conflict identifier (e.g., "conflict-001")
    type: enum         # "contradiction" | "ambiguity" | "overlap" | "gap"
    severity: enum     # "blocking" | "significant" | "minor"
    description: string  # What the conflict is
    sources:
      - path: string
        position: string   # What this source says
        evidence: string   # Quote or reference
    resolution:
      status: enum     # "unresolved" | "resolved" | "deferred"
      decision: string # How it was resolved (if resolved)
      rationale: string # Why this resolution
      authority: string # Who/what made the decision

# Optional: Information gaps identified
gaps:
  - id: string         # Gap identifier
    description: string  # What's missing
    needed_for: string   # Why it matters
    suggested_source: string  # Where to look for this information

# Optional: Notes for synthesis agent
synthesis_notes:
  recommended_structure: string  # Suggested outline for consolidated doc
  tone_guidance: string         # Voice/style notes
  audience: string              # Who the consolidated doc is for
  special_handling:             # Anything unusual about this topic
    - string
```

## Validation Rules

### Required Fields

| Field | Rule |
|-------|------|
| `schema_version` | Must be semantic version string (e.g., "1.0") |
| `topic.id` | Must be kebab-case, 3-50 characters |
| `topic.name` | Must be non-empty string |
| `metadata.created_at` | Must be valid ISO 8601 timestamp |
| `sources` | Must have at least one entry |
| `sources[].authority` | Must be one of: "primary", "secondary", "supplementary" |
| `canonical_sections` | Must have at least one entry |
| `conflicts` | Required but may be empty array |

### Enum Constraints

```yaml
authority:
  - primary       # The definitive source for this content
  - secondary     # Provides supporting/additional detail
  - supplementary # Background or context only

conflict_type:
  - contradiction # Sources say opposite things
  - ambiguity     # Sources are unclear or vague
  - overlap       # Redundant content needs deduplication
  - gap           # Missing information between sources

conflict_severity:
  - blocking      # Cannot synthesize until resolved
  - significant   # Should resolve but can work around
  - minor         # Note for future but doesn't block

resolution_status:
  - unresolved    # Needs human decision
  - resolved      # Decision made, apply it
  - deferred      # Acknowledged but not addressing now

relationship:
  - depends_on    # This section requires understanding of target
  - extends       # This section builds on target
  - supersedes    # This section replaces target
  - related       # General relationship
```

### Business Rules

1. **Single Primary Authority**: Each `canonical_section` must have exactly one `authority_source`
2. **Conflict Resolution Tracking**: All "blocking" conflicts must be resolved before synthesis can proceed
3. **Source Verification**: Every path in `sources` must exist and hash must match current file content
4. **Token Budget**: `extraction_token_count` should be <2000 for efficiency; warn if >3000
5. **Reference Integrity**: All `cross_references.target` values must resolve to valid section/topic IDs

## Example

```yaml
schema_version: "1.0"

topic:
  id: "settings-merge"
  name: "Settings Merge Algorithm"
  scope: "How settings from multiple tiers combine during cem sync"

metadata:
  created_at: "2024-12-25T10:30:00Z"
  updated_at: "2024-12-25T10:30:00Z"
  extractor: "extractor-agent-v1"
  source_token_count: 8500
  extraction_token_count: 1850

sources:
  - path: "{channel_dir}/skills/doc-ecosystem/INDEX.lego.md"
    hash: "a1b2c3d4..."
    authority: primary
    sections_used:
      - heading: "Settings Merge Rules"
        line_range: [145, 210]
  - path: ".ledge/specs/TDD-0042-settings.md"
    hash: "e5f6g7h8..."
    authority: secondary
    sections_used:
      - heading: "Merge Algorithm"
        line_range: [50, 95]
      - heading: "Conflict Resolution"
        line_range: [96, 120]

canonical_sections:
  - id: "tier-precedence"
    heading: "Tier Precedence"
    authority_source: "{channel_dir}/skills/doc-ecosystem/INDEX.lego.md"
    content_summary: "Settings merge follows a strict precedence order: user > team > project > skeleton. Later tiers override earlier ones for scalar values."
    key_points:
      - "Four tiers: skeleton (base), project, team, user (highest)"
      - "Scalar values: later tier wins completely"
      - "Arrays: configurable merge strategy (replace, append, merge)"
      - "Objects: recursive merge with tier precedence at each level"
    code_examples:
      - language: json
        snippet: |
          {
            "skeleton": {"timeout": 30},
            "user": {"timeout": 60}
          }
          // Result: {"timeout": 60}
        source: ".ledge/specs/TDD-0042-settings.md:72"

  - id: "array-strategies"
    heading: "Array Merge Strategies"
    authority_source: ".ledge/specs/TDD-0042-settings.md"
    content_summary: "Arrays support three merge strategies configured per-key: replace (default), append, and merge (union)."
    key_points:
      - "replace: Higher tier array replaces lower tier entirely"
      - "append: Higher tier items added after lower tier items"
      - "merge: Union of items, duplicates removed"
    cross_references:
      - target: "tier-precedence"
        relationship: depends_on

shared_concepts:
  - concept: "tier"
    definition: "A layer of configuration with defined precedence in the merge hierarchy"
    sources:
      - path: "{channel_dir}/skills/doc-ecosystem/INDEX.lego.md"
        treatment: "Defines the four-tier model with examples"
      - path: ".ledge/specs/TDD-0042-settings.md"
        treatment: "Technical implementation of tier loading order"

conflicts:
  - id: "conflict-001"
    type: contradiction
    severity: significant
    description: "Sources disagree on default array merge strategy"
    sources:
      - path: "{channel_dir}/skills/doc-ecosystem/INDEX.lego.md"
        position: "Default strategy is 'append'"
        evidence: "Line 178: 'Arrays append by default...'"
      - path: ".ledge/specs/TDD-0042-settings.md"
        position: "Default strategy is 'replace'"
        evidence: "Line 85: 'replace (default)'"
    resolution:
      status: resolved
      decision: "Default is 'replace' per TDD (implementation source of truth)"
      rationale: "TDD reflects implemented behavior; SKILL.md was aspirational"
      authority: "ecosystem-analyst-2024-12-25"

gaps: []

synthesis_notes:
  recommended_structure: "Start with tier overview, then scalar rules, then array strategies with examples"
  tone_guidance: "Technical but accessible; assume reader knows JSON but not the codebase"
  audience: "Satellite developers configuring their projects"
```

## Usage Notes

### For Extractor Agent

1. Read all source files for the topic
2. Identify which source is authoritative for each concept
3. Extract key information into structured sections
4. Flag all conflicts explicitly - do not silently resolve
5. Keep extraction under 2K tokens by summarizing, not quoting extensively

### For Synthesis Agent

1. Trust the extraction - do not re-read source files
2. Use `canonical_sections` as the document outline
3. Check `conflicts` before proceeding - block on "blocking" severity
4. Apply `resolution.decision` for resolved conflicts
5. Follow `synthesis_notes` for structure and tone

### Staleness Detection

Compare `sources[].hash` against current file hashes. If any differ:
- Re-extraction required for that source
- Check if conflicts or canonical sections are affected
- Update checkpoint to reflect re-extraction need
